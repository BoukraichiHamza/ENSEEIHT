1.     Objectif

Dans ce tp, on gère des données répliquées en utilisant de la diffusion de messages (write update).
Dans une première étape (dup.zip), les clients se connaissent et s'envoient les mises à jours (write update) simplement en s'envoyant des messages UDP. Le système est composé des classes suivantes :

  • Launcher/launcherImpl est le serveur qui reçoit les abonnements des clients (par RMI). Un client qui s'abonne donne son nom de machine et son numéro de port, ainsi qu'une référence RMI pour être rappelé. Le serveur accumule les abonnements dans un objet group (Group/GroupImpl), puis rappelle les clients pour leur donner une copie de cet objet group.
  • Client/ClientImpl est le programme exécuté par un client. Il s'abonne auprès du serveur et peut être rappelé par RMI par le serveur pour recevoir l'objet group. Il crée 2 threads : Appli qui implante le code de l'application (mettre à jour des valeurs dans la copie locale des données partagées par réplication), et Daemon qui reçoit les mises à jour et les répercute dans la copie locale des données partagées.
  • Group/GroupImpl est la classe qui permet d'envoyer des messages au groupe des clients. Son état inclus les destinations des clients (nom de machine / numéro de port) pour envoyer des messages aux clients.
  • Target est juste une classe définissant les destinations (nom de machine / numéro de port). - Update est la classe définissant le format des messages échangés (des messages de mise à jour des données).

2.     Etape 1

La première étape du TP est d'exécuter ce code :

    • java LauncherImpl : permet de lancer le serveur
    • java ClientImpl <numero de port du client> : permet de lancer un client. Il faut juste avoir la variable registryHost mise à jour dans ClientImpl.java pour indiquer la machine où se trouve le serveur.

On peut tester en local ou en réparti.
Launcher ouvre une fenêtre avec 2 boutons : Go lance l'exécution et Print affiche en console l'état des données.
Vous devez observer que les données peuvent devenir incohérentes.
Pour cela, il faudra si nécessaire lancer l'application en réparti.


3.     Etape 2

Dans la deuxième étape, on demande d'implanter un séquenceur pour garantir la cohérence des données partagées par réplication.
Ceci consiste simplement à :

    • rajouter une méthode getTicket dans Launcher pour que les clients puissent demander un ticket (numéro de séquence).
    • modifier Appli pour qu'elle demande un numéro de séquence avant une écriture et qu'elle n'applique une écriture que si elle est à jour dans l'application des modifications (l'application mémorise le numéro de la dernière modification effectuée), sinon elle doit attendre.
    • modifier Daemon pour que lorsqu'on reçoit une modification, si c'est la prochaine, on l'applique et on essaie d'appliquer celles qui étaient en attente, sinon on la met en attente. Et on réveille l'application si elle est à jour des modifications.

On peut alors vérifier qu'on a une gestion cohérente des données.
